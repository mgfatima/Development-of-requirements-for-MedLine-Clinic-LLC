# МедЛайн Клиника — разработка требований и проектирование ИС (Requirements & IS Design)

Учебный/портфолио-проект по курсу **«Управление требованиями и проектированием ИС»**: подготовка полного комплекта требований и проектных артефактов для информационной системы клиники, которая автоматизирует **онлайн-запись пациентов**, **управление расписанием врачей**, **ведение электронной медкарты** и **уведомления**.

---

## Контекст и проблема
В клинике запись ведётся через несколько каналов (телефон/сайт/мессенджеры), данные фиксируются разрозненно и частично вручную. Это приводит к:
- накладкам и конфликтам в расписании,
- дублированию записей,
- росту нагрузки на администраторов,
- сложности быстрого доступа к медицинским данным.

---

## Цели и ограничения проекта
**Бизнес-цели (эффект внедрения):**
- снизить нагрузку на администраторов на **25–30%**,
- сократить время записи/ожидания пациента,
- повысить лояльность за счёт удобной онлайн-записи и напоминаний.

**Ограничения:**
- соответствие ФЗ-152 (персональные данные) и ФЗ-323 (медицинские данные/врачебная тайна),
- пилотная версия — до **5 месяцев**,
- бюджет — до **13 млн руб.**

---

## Пользователи и роли
- **Пациенты:** онлайн-запись, уведомления, просмотр истории посещений
- **Администраторы:** управление расписанием, контроль загрузки, обработка заявок
- **Врачи:** доступ к электронной медкарте, результатам и назначениями
- **Руководство:** аналитика загрузки и KPI

## Объём решения (ключевые функции)
### Functional Requirements (пример состава)
- **FR1:** интеллектуальная онлайн-запись пациента (выбор врача/специализации/слота, предотвращение конфликтов)
- **FR2:** управление расписанием врачей (редактирование, блокировки, контроль загрузки)
- **FR3:** электронная медицинская карта пациента (история посещений, назначения, результаты)
- **FR4:** уведомления и напоминания (подтверждение записи, напоминания, изменения)

### Non-Functional Requirements (FURPS+)
- **Usability:** запись ≤ 3 шага, единый UI, адаптивность
- **Reliability:** доступность, восстановление после сбоев, SLA
- **Performance:** время отклика, параллельная нагрузка, скорость открытия медкарты
- **Supportability:** мониторинг/логирование, масштабируемость, обновления
- **Security:** RBAC/2FA (для расширенных ролей), аудит, шифрование, соответствие регуляторике

## API (пример)
Ключевой сценарий: **создание записи на приём**  
`POST /api/v1/appointments`

В репозитории есть:
- описание запроса/ответа,
- примеры JSON,
- коды ошибок (400/401/403/409/500),
- OpenAPI (YAML), готовый для Swagger Editor / Swagger UI.

---

## Модель данных (концептуально)
База данных: **PostgreSQL**  
Ключевые сущности: `patient`, `doctor`, `service`, `appointment`, `payment`, `clinic_branch`  
Основная сущность процесса: `appointment` (запись пациента на приём) со статусами: `planned/confirmed/completed/cancelled/no_show`.
